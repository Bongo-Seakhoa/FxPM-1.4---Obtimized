Methodology Alignment

Data Workflow

The active workflow is coherent: latest 300,000 M5 bars, oldest 50,000 M5 bars as historical_stress_audit, newest 250,000 M5 bars as the active universe, and Stage 3 risk/governance selection on the freshest Stage 2 surface.

Walk-Forward

Walk-forward should remain an offline audit report, not a mandatory promotion layer. Adding it between Stage 2/Stage 3 and live implementation would reduce freshness and conflict with the live-relevance goal.

Backtest Balance

A larger notional backtest balance is acceptable for strategy discovery. The live layer is responsible for broker feasibility, minimum lot, margin, and sizing constraints.

Findings

F1 - Live Eligibility Does Not Enforce Expiry Or Artifact Due Status Resolved

ConfigLedger.has_valid_config() rejects expired configs, and should_optimize() detects artifact fingerprint drift. But PortfolioManager.get_validated_configs() returns configs based on is_validated only.

A direct probe confirmed that an expired config is rejected by the ledger but still returned by get_validated_configs(). Since live trading uses that surface, expired configs can remain live-tradeable.

Implemented: live trading now uses get_live_eligible_configs(), blocks expired/no-winner/artifact-drifted configs under live_artifact_drift_policy = "block", and status now reports live eligibility with BL markers. Artifact invalidation ignores volatile ledger metadata.

F2 - Active High-Risk Ledger Is Not Final Or Fully Propagated High

The active ledger contains only 14 of 62 symbols, and --status reports 59 symbols needing optimization. Most configured symbols are marked due because the artifact fingerprint changed.

Recommendation: do not judge final PM quality from the current ledger. After deliberate optimization, verify symbol coverage, artifact contract consistency, validation evidence, data-window fingerprinting, ledger status, and robustness evidence.

F3 - Git/Deployment Boundary Is Not Clean High

Core local files are untracked by git, including pm_storage.py, pm_order_governance.py, pm_dashboard/ledger.py, pm_configs_high_risk.json, and several tests.

Recommendation: before live deployment or handoff, decide what is generated and what is source. Track source modules and tests, and deliberately provision or track the active production ledger.

F4 - Margin Guards Do Not Normalize Non-Finite Values Resolved

_safe_account_margin_level() converts values with float() but does not reject NaN or inf. Those values can avoid both missing-margin and normal threshold branches.

Implemented: non-finite margin level, free margin, and required margin now fail closed with SKIPPED_MARGIN_UNAVAILABLE where trading would be unsafe.

F5 - Same-Symbol Combined Risk Still Uses Stale Sweep Snapshot Resolved

The exact duplicate-position guard now performs a fresh broker read, but the later same-symbol combined-risk check still receives the original sweep snapshot. A same-symbol/different-magic position opened after the sweep could be missed.

Implemented: the fresh symbol-level broker positions are now passed into _check_symbol_combined_risk_cap(), with regression coverage.

F6 - Dashboard Defaults Hide Margin And Low-Balance Skip Actions Resolved

Important actions such as SKIPPED_MARGIN_REQUIRED, SKIPPED_MARGIN_BLOCKED, and BLOCKED_MIN_LOT_EXCEEDS_CAP may be recorded but hidden by dashboard display filters.

Implemented: dashboard defaults now include margin and minimum-lot feasibility actions/prefixes while keeping noisy NO_ACTIONABLE_* entries excluded.

F7 - Storage/Cache Is Healthy, But State Semantics Need Tightening Resolved

The cache/storage design is in a good zone: 4 GB quota, current use well below quota, observe-only cleanup, resample-cache telemetry, and storage tests. The state file can still mix current housekeeping with stale live-sweep fields.

Implemented: storage state now carries state_updated_at and freshness labels for sweep/housekeeping fields. The 4 GB cache cap remains unchanged.

F8 - Production Retrain Mode Is Notify, Not Auto Medium-Low

production_retrain_mode = notify is valid for manual control, but it means due symbols are announced rather than automatically refreshed.

Recommendation: keep notify only if manual retrain control is intended. Use auto only when the machine, broker connection, storage, and time budget are ready.

Non-Issues Confirmed

  • The active-recent M5 workflow is not broken by not being a traditional academic train/validation/holdout design.
  • historical_stress_audit is the correct name for the oldest 50,000 M5 bars.
  • Walk-forward should remain an offline audit, not a mandatory live-promotion gate.
  • Larger backtest notional balance is acceptable for strategy discovery.
  • Low-balance live rejections are expected and should not be over-interpreted.
  • Regime live/optimization logic is now aligned around REGIME_LIVE at decision time.
  • Backtester and live execution now share the strategy trade-intent exit surface.
  • The 4 GB cache setting is reasonable; shrinking cache is not currently indicated.
