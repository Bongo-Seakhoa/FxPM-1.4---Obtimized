FundedNext Stellar Lite MT5 Challenge Customization Proposal

Prepared on March 31, 2026 from current official FundedNext sources plus a local audit of this PM codebase and config.json. Audited and updated with deep codebase verification and rule cross-check.

Verdict: MT5-compatible, but not with the current generic broad-market defaults
Best starting posture: FX-focused, lower live risk, equity-based sizing
Config governance: core PM is config-led, dashboard/auxiliary layer is only partially so
Important date-specific note: as of March 31, 2026, FundedNext's public plan page shows $6K as the smallest current public plan size, but FundedNext's Stellar Lite Help Center articles still document $5,000 examples and thresholds. For this proposal I assume you either have a legacy/current 5K MT5 Stellar Lite challenge credential, or you want the same profile adapted to that account size. The real source of truth is the balance shown in your actual challenge terminal. If the broker account is $6,000 instead of $5,000, scale the dollar amounts proportionally.

Executive Recommendation

The PM is technically viable for an MT5 Stellar Lite challenge, but I would not run the current generic configuration on a small FundedNext evaluation. The current defaults are built for a broad multi-asset PM and are too permissive for a challenge account whose official hard limits are tighter and whose rule enforcement explicitly penalizes gambling-style exposure.

My recommendation is to run a challenge-specific profile with a smaller symbol universe, equity-based live sizing, materially lower per-trade risk, no same-symbol stacking by default, and a stronger operational assumption that automation on MT5 is allowed only under the dedicated EA/VPS policy.

Best practical profile: FX-only or FX-dominant, M15/M30/H1/H4, 0.35% live risk per trade, 0.80% same-symbol combined cap, equity risk basis, and notify rather than unattended auto-retrain while the challenge is live.

Why the Current Generic Config Is Too Hot

  • The current root config still uses 1.0% risk per trade in both pipeline and position.
  • The live default risk basis is balance, not equity, which is wrong for a prop challenge where floating loss counts toward breach.
  • The PM currently allows a 3.0% same-symbol combined risk cap, which is too large relative to a 4% daily limit.
  • The symbol list is very broad: FX, metals, indices, energies, crypto, and exotics. On a 5K prop challenge, that creates unnecessary spread, swap, min-lot, and commission-model risk.
  • The live path still has no hard account-wide aggregate exposure cap; it is safer than before, but not challenge-tight.

Instrument and Cost Modeling Guidance

The PM's live sizing is much stronger than it used to be because MT5 tick-value and contract-size data are used in the live path. However, the research/backtest side still depends heavily on config.json and broker_specs.json for realistic cost modeling.

What I would do before optimizing this challenge profile

  • Connect to the actual FundedNext MT5 account and export fresh broker specs into broker_specs.json.
  • Keep broker_specs_path pointed at that file in config.json.
  • Restrict the challenge universe to FX until the broker specs and commission assumptions are confirmed for other asset classes.

Why this matters

  • The dedicated FundedNext commission article updated after January 12, 2026 says crypto and metals commissions are now price-based formulas, not the simple flat values currently implied by this PM's root config for some symbols.
  • The current PM root config has zero commission entries for crypto, which is not aligned with the current FundedNext commission article.
  • On a 5K challenge, cost-model mismatch is not noise. It can change which low-timeframe setups survive validation.

PM-to-Broker Compatibility Audit

What is solid today:

  • The MT5 path resolves broker symbols rather than assuming naked symbols always trade as-is.
  • The live order path uses broker symbol info, stop-level checks, and order_check() before order_send().
  • Real MT5 contract/tick math is used for live loss-per-lot where possible.
  • Live sizing has a spread-aware execution overlay and fail-closed behavior when account or position snapshots are unavailable.

What still matters for a challenge:

  • If the raw risk-based size is below broker minimum lot, the trade is skipped (confirmed at pm_position.py:379-417). This is safe, but it means some valid setups with very tight stops will never execute on a 5K account.
  • The PM has a same-symbol risk cap, but not a strict whole-account concurrent-risk cap.
  • Current tradability filtering blocks disabled symbols, but not every nuanced trade-mode restriction early.

Critical Code Gaps for Prop Firm Use

These gaps must be addressed before running on a real FundedNext challenge.

GapCurrent StateProp Firm RiskCode Location
No daily loss halt The PM has zero daily P&L tracking or halt mechanism. Trading continues regardless of intraday losses. Could blow through the 4% daily loss limit without any automated protection. This is the #1 operational risk. Not implemented anywhere in pm_main.py
No account-wide concurrent risk cap Only per-symbol cap exists (max_combined_risk_pct = 3.0%). No aggregate check across all open positions on all symbols. Theoretically 10+ symbols × 3% each = 30%+ aggregate exposure. No mechanism stops this. pm_main.py:909-1021 (symbol-level only)
No max concurrent trades hard limit ExposureCorrelationScalar reduces risk at 10 positions but never blocks trades. It tapers to 40% of base risk. Scalar reduces but doesn't prevent trades. You could open 50+ positions at reduced risk. pm_enhancement_seams.py:98-112
No margin utilization tracking The PM does not track or cap total margin usage as a percentage of equity. FundedNext enforces a 70% max cumulative margin cap. The PM has no awareness of this. Not implemented
SL confirmation not verified The PM sends SL via order_send() but does not confirm the SL was accepted by the broker after the order fills. FundedNext's 3-minute SL rule: if SL is missing for 3 min, the trade is treated as 100% risk. A rejected SL could trigger this. pm_main.py (execution path)

What the PM does protect against (confirmed working)

  • Per-symbol risk cap: max_combined_risk_pct is enforced in the live path (pm_main.py:909-1021). Sums all open positions on the same broker symbol and blocks if over limit.
  • Min-lot safety: If calculated volume is below broker min_lot, the trade is skipped (not forced up). Confirmed at pm_position.py:379-417.
  • risk_basis = "equity": Properly reads account.equity when configured. Confirmed at pm_main.py:1821-1826.
  • Spread filter (G5): Active by default. Blocks trades during spread spikes or insufficient edge. Confirmed at pm_main.py:1834-1865.
  • Drawdown risk reduction: DrawdownPositionScalar reduces sizing as drawdown deepens (to 25% of base at 20% DD). Not a halt, but a soft brake.

Why I Exclude Crypto, Metals, Indices, Energies, and Exotics Here

  • Crypto: commission model has moved, and the current config is not the right truth source for those costs.
  • Metals / indices / energies: wider spread regimes, more violent gap behavior, and higher min-lot sensitivity matter more on a 5K challenge.
  • Exotics: spreads and swaps are more likely to create floating-loss surprises against the daily limit.
  • Broad multi-asset scope: the PM can technically do it, but the challenge objective is survival and clean progression, not proving the widest possible asset coverage.

If you later want a second-stage challenge profile, the first non-FX addition I would test is XAUUSD only, and only after refreshing broker specs and confirming the current FundedNext commission behavior on that specific MT5 server.

Short answer: the PM core is config-primary, but the codebase is not a perfect end-to-end single-source-of-truth system yet.

AreaStatusWhat I found

Root config rewrite risk Low I did not find a normal runtime path that rewrites root config.json. The code writes separate artifacts such as pm_configs.json, broker_specs.json, and dashboard config files.

Dashboard Partial only The dashboard still assumes default root filenames and default data/pm_outputs folders in several places, and its MT5 maintenance path does not consume the PM's root mt5 section.

This includes: target_annual_vol=0.10, max_positions=10, kelly_fraction=0.25, max_dd_pct=20.0, min_edge_mult=1.5, spike_mult=2.0. None of these are configurable.
Silent key dropping Design choice with side-effects _filter_dataclass_kwargs() in pm_main.py:49-52 silently drops any config.json key that doesn't match a dataclass field. Typos in config keys produce no warning - the system falls back to hardcoded defaults without telling you.
getattr() fallbacks in live path Hardcoded safety nets Live trading code uses getattr(self.pipeline_config, 'max_combined_risk_pct', 3.0) and similar patterns. If a config field is misspelled or missing, you silently get the hardcoded default (3.0%, 1.0%, 5.0%) instead of an error.

What this means in practice

  • For the trading core: yes, use config.json as the operational source of truth.
  • For the dashboard: do not assume perfect config propagation if you use nondefault root paths or alternate config files.
  • For this challenge: keep the PM on the default root config path if you want the fewest surprises.

Challenge-Specific Operational Rules I Would Personally Follow

Before Starting

  • Confirm the EA is not on FundedNext's prohibited EA list (it's custom-built, so it shouldn't be, but verify trade-pattern similarity is not an issue).
  • Pay for the relevant EA usage fee and/or VPS add-on if FundedNext requires it for this account.
  • Refresh broker_specs.json from the actual challenge MT5 login before the first optimization pass.
  • Verify the SL is actually accepted on the broker after order_send() - FundedNext counts trades with no SL after 3 minutes as 100% risk.

During the Challenge

  • Use one stable device or one dedicated VPS/IP. Do not hop countries or rotate endpoints.
  • Do not run the same PM profile across multiple FundedNext accounts unless the strategies are genuinely distinct and policy-compliant.
  • Run the challenge with the narrowed FX-only symbol universe only.
  • Use production_retrain_mode = "notify" while the challenge is active unless the full auto-retrain loop has already been rehearsed end-to-end on a noncritical account.
  • Do not let the PM operate near the raw 4% daily breach line. Treat 1.0%-1.5% open risk across the book as the practical ceiling unless you add a true account-wide cap in code.
  • Manually monitor daily P&L - the PM has no daily loss halt mechanism. Consider adding an external watchdog or cron job that closes all positions and stops trading if daily loss exceeds 2%.
  • Keep margin utilization below 30% (FundedNext recommends 20-30%, hard cap at 70%).

After Passing (Funded Phase)

  • Weekend holding is NOT allowed on funded accounts - close all positions before Friday market close. The PM does not enforce this automatically.
  • The formal 3% max risk at any given time rule applies. The PM's current per-symbol cap of 3.0% only covers one symbol; aggregate exposure across all symbols is not capped.
  • Expect the first payout after 21 days, then every 14 days. The subscription fee is refunded with the 3rd withdrawal.

Final Judgment

My honest signoff: the PM can be made suitable for an MT5 FundedNext Stellar Lite challenge, but only through a challenge-specific configuration profile and addressing the critical code gaps identified above. The current generic root config is too broad and too permissive for a small evaluation account.

If I were preparing this system for a real challenge attempt, I would do exactly this:

  1. Confirm the account is truly MT5 and not subject to the current U.S. MetaQuotes restriction.
  2. Export fresh broker specs from the live challenge account.
  3. Replace the current symbol list with the recommended FX-only set.
  4. Set both backtest and live risk to 0.35% and switch live sizing to equity.
  5. Disable same-symbol stacking by default for the challenge.
  6. Run in notify retrain mode, not unattended auto mode, until the challenge is passed.
  7. Add a daily loss halt mechanism - either in code or via an external watchdog that kills trading if daily loss exceeds 2%.
  8. Add an account-wide aggregate risk cap - sum all open risk and block new trades above a configurable ceiling (e.g. 2.0% total).
  9. Verify SL acceptance after order fill - confirm the broker actually applied the stop, don't just assume order_send() success means SL is live.

Overall proposal grade: high-confidence on config recommendations and rule compliance, but two critical code gaps (daily loss halt and account-wide risk cap) must be implemented before running on a real challenge.
